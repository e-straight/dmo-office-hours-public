What are the terminal and CLI?

A terminal is a text-based workspace for running commands. A CLI, or command-line interface, is a program you control by typing those commands. GitHub Copilot CLI runs inside your terminal, next to tools such as Git, your test runner, build commands, logs, and project files.

terminal — sample-project
$cd sample-project && copilot GitHub Copilot CLI · trusted directory > Explain the failing signup test. Reading @src/signup.ts and test output... Root cause: malformed email values bypass validation. ✓ Proposed fix · ✓ Targeted test passed
Commands stay closeGit, builds, logs, tests, and Copilot share one workspace.
Context is explicitMention files, issues, images, and directories directly.
You keep controlApprove tools, inspect diffs, and verify the result.

It is not an either/or choice: use the CLI for terminal-native execution and the app for visual orchestration. Pick the surface that best matches the task and how you like to work.

1 Install

Choose the option that fits your machine:

brew install --cask copilot-cli
winget install GitHub.Copilot
npm install -g @github/copilot

The npm option requires Node.js 22 or later. See the official installation guide for the install script and platform details.

2 Launch and sign in

  1. Open a terminal in a repository or trusted folder.
  2. Run copilot.
  3. Follow the browser sign-in flow. Use /login if you need to sign in later.
cd path/to/your-project && copilot
Live help wins: Copilot CLI evolves quickly. Run /help for the commands supported by your installed version.

Optional: Connect Microsoft 365 with Work IQ

Work IQ lets Copilot use permitted context from your email, meetings, Teams messages, documents, and workplace directory. Inside Copilot CLI, add the official plugin marketplace once, then install Work IQ:

/plugin marketplace add github/copilot-plugins
/plugin install workiq@copilot-plugins

Restart Copilot CLI after installation, then try: What are my upcoming meetings this week?

Organization setup required: Work IQ requires Node.js, an assigned usage-based billing plan, and Microsoft Entra administrative consent before it can access Microsoft 365 organization data.

See Microsoft's Work IQ CLI installation guide for current prerequisites, setup, and privacy details.

3 Run your first task

ExploreAsk how the project works before changing it.
ScopeName the files, constraints, and success condition.
ApproveReview each requested tool or command.
VerifyAsk Copilot to run the relevant tests.

Good first prompts

Explain this repository to a new contributor. Identify the main entry points, how to run it, and where tests live. Do not change files.
Find one small, low-risk improvement in this project. Explain the proposed change first, wait for my approval, then implement it and run the most relevant test.
Investigate why the current test is failing. Identify the root cause before editing, make the smallest safe fix, and rerun the targeted test.

4 Give it useful context

  • @path/to/file mentions a file.
  • #123 mentions a GitHub issue or pull request.
  • ! git status runs a shell command yourself.
  • Drag, paste, or mention an image for UI work.
Using @src/auth.ts and #123, explain the bug, propose the smallest fix, and list the tests that should prove it works.

5 Use the right mode

SteerStandardExplore, ask questions, debug, and make small interactive changes.
AlignPlanDesign multi-file features and refactors before any code is written.
ExecuteAutopilotRun a bounded, reversible task with reliable checks and less supervision.

Press Shift+Tab to switch modes. Plan mode waits for approval before implementation.

6 Prompt with a definition of done

A strong task usually has four parts:

GoalWhat outcome do you need?
ContextWhich files, issue, error, or behavior matters?
ConstraintsWhat must stay unchanged? What risks matter?
DoneWhich tests, checks, or output prove success?
Add validation to the signup endpoint in @src/routes/signup.ts. Preserve the existing response shape, reuse the project's validation helpers, add tests for empty and malformed email values, run the targeted test suite, and summarize the diff.

Safety essentials

  • Read permission prompts; do not reflexively choose "always allow."
  • Review diffs and command output before committing or pushing.
  • Never paste or commit secrets, tokens, or private keys.
  • Use /reset-allowed-tools to clear session approvals.
  • Use /diff to inspect current changes.
High-risk actions: slow down for installs, migrations, deletions, force operations, deployments, and commands that access credentials or production systems.

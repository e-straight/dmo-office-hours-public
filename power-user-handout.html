1 Think “control center,” not “chat box”

The beginner move is one prompt against one folder. The power-user move is several scoped agents, each with a clear plan, visible artifacts, and a review path back into GitHub.

2 High-value power workflows

Agent fleet orchestration

Treat sessions like a small fleet: each agent gets one job, one success condition, one workspace, and one review surface. Use a canvas to track the fleet so you can steer at the portfolio level instead of babysitting individual chats.

Docs: Working with agent sessions

Builder signal: Evan Boyle highlights multi-session orchestration and persistent cloud sessions, then calls /orchestrate one of the app's killer features. GitHub’s Evan deep dive connects automations, MCP integrations, and custom skills to repeatable background work.
I want to run an agent fleet for this repo: bug-fix, test-hardening, docs, dependency audit, and independent reviewer. Create a fleet plan with session goals, workspace choice, model choice, risks, handoffs, and done criteria.

Plan-first issue work

For complex or production-facing changes, make Copilot propose the path before it edits files.

Create a plan for fixing issue #123. Do not change files yet. Identify risks, tests to run, and the smallest safe implementation path.

Persistent cloud sessions

Use cloud sandboxes when work should keep its own saved state, run away from your laptop, or resume later. Good fits: heavy test suites, environment setup, dependency upgrades, cross-repo investigations, or long-running refactors.

3 Fleet dashboards and canvases big unlock

Use canvases when the work needs structure: boards, dashboards, checklists, plans, docs, diffs, or decision records. They make agent work easier to steer than a long chat transcript.

Docs: Working with canvas extensions

Builder signal: James Montemagno shows a personal “James Hub” canvas for workload management, priorities, and time allocation, plus a Copilot app canvas for app submissions and tooling workflows.
Create an agent fleet dashboard canvas that tracks each session's owner, workspace, mode, model, current hypothesis, last action, blocker, next decision, and PR / diff link.
Create an agentic kanban canvas for this project with backlog, in progress, blocked, ready for review, and done. Include actions to add, move, and assign cards.
Create a launch command-center canvas with milestones, risks, owners, open questions, linked issues, and next actions.
Create an issue triage dashboard that groups recent issues by theme, severity, customer impact, and recommended next action.

4 Route the right model to the right job

Advanced users should stop treating “the model” as one setting. Pick mode, model, reasoning effort, and provider based on risk and complexity. If you use BYOK, the app can add models from external providers and local / OpenAI-compatible endpoints.

5 Automate research and recurring work

Automations are strongest when the source set is stable and the output shape is repeatable. Be explicit about sources, citations, open questions, and what not to infer.

Docs: Using automations

Builder signal: GitHub’s Evan interview covers automations and custom skills for scheduled or recurring workflows. Pierce Boggan also points to automations that cut context-switching and shares a personal automation example built in the app.
Every Friday at 9am, research what changed this week on <topic> using the available project sources, then draft a one-page brief with citations, open questions, and recommended follow-ups.
Every Monday at 8am, summarize last week's merged PRs and open issues for this repo, grouped by customer impact, risk, and follow-up needed.
On demand, generate release notes from merged PRs since the last tag, then flag anything that needs human review before publishing.
When a new issue matching label:customer-impact is opened, triage it, identify likely owner areas, propose labels, and draft a first response without posting it.
Connector note: Slack / Microsoft 365 may work in normal chat before they work reliably inside Automations. If a connector-backed automation fails, run that prompt in chat for now.

6 PR follow-through and verification loops

Use agents for mechanical review follow-up, but keep humans in control of architecture, product tradeoffs, and final approval. For risky work, use a maker/checker pattern.

Docs: Using the rubber duck agent · Working with agent sessions

Builder signal: James Montemagno calls Agent Merge a game changer for creating the PR, running actions, running code review, and preparing it for merge. Pierce Boggan says Agent Merge makes it easier to land PRs from coding agent sessions.
Address the review comments on this PR, investigate any failing checks, and summarize what changed before I approve anything.
Act as an independent verifier for this branch. Do not edit files. Review the plan, diff, tests, security implications, rollback path, and missing edge cases. Return only high-confidence issues.
Create a merge-readiness checklist for this PR: CI status, test coverage, docs, migrations, rollout risk, observability, and rollback plan.

7 Write your first skill

A skill teaches Copilot a repeatable workflow: the goal, inputs, steps, examples, guardrails, and success criteria. Good first skills are team workflows you already repeat by hand.

Docs: Adding agent skills · Customizing the app

8 Habits that make advanced usage reliable

Builder signal: The strongest habit theme across the builder posts is intentional architecture: isolated worktrees, /orchestrate, and deliberate cloud-vs-local decisions from Evan Boyle; visible canvases and Agent Merge loops from James Montemagno and Pierce Boggan; Kyle Daigle’s daily-driver usage across coding, automations, rubber ducking, skills, MCP, autopilot, and remote steering; and the .NET framing of sessions, canvases, and automations that make AI work like a teammate, not a textbox.
  • Scope before launch. Define the goal, risks, constraints, and done criteria before starting a long-running session.
  • Match autonomy to risk. Use plan-first for high-stakes work; use more autonomy only for narrow, reversible, well-tested tasks.
  • Design fleets intentionally. Give each agent a unique job; avoid five agents all solving the same problem unless you are deliberately comparing approaches.
  • Use cloud sessions selectively. Great for resumable, compute-heavy, or cross-device work; be mindful of org policy, billing, secrets, and tool access.
  • Route models deliberately. Use cheaper / faster models for routine edits; spend reasoning budget on design, debugging, migrations, and review.
  • Prefer visible artifacts. Ask for a plan, canvas, checklist, or diff summary so humans can steer quickly.
  • Separate maker and checker. A second review session is cheap insurance for production-oriented work.
  • Standardize what repeats. If the same prompt, checklist, or review policy keeps coming up, turn it into a skill, automation, shared instruction, or extension.
  • Stay connected. Share wins and reusable prompts in #dmo-ai-first.

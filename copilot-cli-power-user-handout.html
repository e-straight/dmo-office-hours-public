Quick start Power user
Power-user principle: give Copilot the smallest useful trust boundary, a concrete definition of done, and an independent verification step. More autonomy should come from better constraints, not less oversight.

Remote is not cloud execution: the VM must remain online, the CLI session must still be active, and the working directory must be a GitHub-hosted repository. Use /remote on or copilot --remote; use /remote off when finished. If the machine should not remain running, consider /delegate instead.

3 Design permission boundaries

Pre-allow routine, narrow tools and continue prompting for risky actions.

copilot --allow-tool='shell(git:*)' --deny-tool='shell(git push)'
copilot --allow-tool='shell(bun test:*)' --allow-tool='write'
  • Use /reset-allowed-tools when the task changes.
  • Use /add-dir only for paths the task truly needs.
  • Reserve /allow-all for isolated, disposable environments.
  • Enterprise policy may disable broad approval options.
Permission matching is not a sandbox. Review destructive commands, URL access, credential use, and writes outside the repository even when a pattern appears narrow.

7 Work across repositories

Start from a trusted parent directory or add only the extra repositories required:

/add-dir /absolute/path/to/shared-library
/list-dirs
Trace the authentication contract across @/path/api, @/path/web, and @/path/sdk. Do not edit. Identify incompatible assumptions and propose an order of operations that keeps each repository releasable.
Coordinate commits: ask for a repository-by-repository plan, compatibility strategy, and test matrix before making cross-repo edits.

8 Separate maker and checker

The implementing agent has context and momentum, but also shares its own assumptions. Use a fresh reviewer or a different model when risk warrants it.

/review Review the current branch against main. Report only high-confidence correctness bugs, regressions, and missing tests. Ignore style. Do not edit files.
Act as an independent verifier. Reconstruct the requirements from the issue and diff, run the smallest relevant test set, inspect error paths and rollback behavior, and identify anything that prevents merge. Do not change files.
/security-review Focus on exploitable trust-boundary, authorization, injection, secret-handling, and unsafe-deserialization issues introduced by this diff.
Reliable landing sequence: targeted tests → diff review → independent review → full required checks → PR → CI/review follow-through.

9 Script repeatable one-shot work

Use programmatic mode when inputs, permissions, and output shape are predictable:

copilot -p "Summarize the current diff and list release-note-worthy changes."

For automation, make the prompt deterministic, constrain allowed tools, and require machine-checkable output. Avoid broad autonomous write access in CI.

Provider choice: use /model for the installed GitHub-hosted catalog, or copilot help providers for BYOK setup. Copilot CLI supports OpenAI-compatible endpoints—including local Ollama, vLLM, and Foundry Local—plus Azure OpenAI and Anthropic providers.
  • Custom models must support streaming and tool calling; a context window of at least 128k is recommended.
  • Use COPILOT_OFFLINE=true for isolated operation, but full isolation requires a local or on-premises provider.
  • Subagents inherit the configured provider; /delegate still transfers work to GitHub's cloud agent.
